# Reflexão Pessoal: Desafios e Aprendizados com Git

## O que foi difícil
A adaptação ao Git exigiu bastante tentativa e erro, tive que entender como o Git absorve os arquivos em constrate a como o Visual Studio Code os exibe. O primeiro obstáculo surgiu na criação da estrutura de pastas do repositório. Ao criar as pastas necessárias para a atividade proposta, o VS Code exibiu as pastas de forma compactada, o que me fez acreditar que a estrutura estava errada. Descobri, após pesquisa, que o Git ignora pastas vazias e eu precisei criar arquivos `.keep` para forçar o rastreamento. Esse foi o primeiro estranhamento sobre a utilização da ferramenta.

No entanto, a minha maior dificuldade foi a simulação do conflito de merges. Eu achava que bastava alterar a branch `main` e tentar um `git pull` para gerar a colisão, mas o Git resolveu a situação com um fast forward, atualizando meu repositório local. Entender o porquê disso não ter falhado exigiu, também, bastante pesquisa. Entendi que o conflito não nasce de apenas "códigos diferentes", mas de linhas do tempo divergentes. Foi necessário criar duas realidades a partir do mesmo momento passado (Branch A e Branch B), para que o Git declarasse sua incapacidade de mesclar as alterações da mesma linha. Após resolver a colisão no arquivo de texto, tive dificuldades novamente ao tentar finalizar o merge, pois o arquivo `REGISTRO.md` que criei estava como "untracked". Esses problemas que aconteceram no meu código exigiram tentativa e erro para serem resolvidas.

## O que ficou claro
O uso do atalho `-am` nos commits foi um conceito que no início tive dificuldade, mas depois consegui fazer. Inicialmente, eu não entendia por que os comandos do Git falhavam em algumas situações. Ao questionar o funcionamento dessa linha, entendi a divisão de responsabilidades: o parâmetro `-a` varre os arquivos que o Git já rastreia e os adiciona. Isso explica por que o atalho falhou com o `REGISTRO.md` durante a resolução de conflitos, já que ele era um arquivo novo. A diferença entre um arquivo modificado e um arquivo não-rastreado fez sentido após essa dificuldade.

Além disso, os commits convencionais ficaram claros. Ao separar a criação da lógica de validação (`feat`) da adição do código que exercita essa lógica (`test`), eu percebi que a árvore de commits deixa de ser um registro desorganizado e passa a ser uma linha do tempo otimizada, onde a intenção da programação é esclarecida antes mesmo de se ler o código.

## O que ainda é confuso
Um ponto de confusão minha é a escalabilidade desse fluxo para um cenário de trabalho em equipe. Executei os merges e Pull Requests de forma totalmente controlada e trabalhando sozinho. Ainda tenho dúvidas sobre o que acontece caso, enquanto estou desenvolvendo uma nova funcionalidade na minha branch, a `main` for drasticamente alterada por outros colegas no repositório remoto? Como mantenho minha branch atualizada sem gerar uma reação em cadeia de conflitos? Nessa atividade, fiz o `merge` funcionar, mas dominar a fluidez entre branches simultâneas em um projeto grande é algo que precisarei exercitar mais para trabalhar de maneira concisa com os demais programadores.
